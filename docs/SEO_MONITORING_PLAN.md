# SEO monitoring plan

Reviewed: 14 July 2026

No paid monitoring service or scheduled automation is authorised by this document.

## Weekly for the first four weeks after an approved merge

- Load the homepage, About, Thinking and Facts over HTTPS.
- Confirm sitemap, robots, feeds, facts and verification file return successfully.
- Review the latest GitHub Pages deployment.
- Check Search Console Page Indexing and URL Inspection for priority routes.
- Check Bing URL Inspection and sitemap status.
- Search branded queries and record titles, snippets, images and stale legacy results.
- Review structured-data enhancement/error reports.
- Confirm old unsupported copy is disappearing from search and retrieval surfaces.
- Check the public email link and LinkedIn link without sending a test message unless authorised.

## Monthly

- Export Search Console query/page performance.
- Review Bing search and backlink reports.
- Count indexed canonical pages and any indexed compatibility routes.
- Review click-through rates without rewriting titles solely for clicks.
- Check portrait and article-image visibility.
- Audit external mentions and backlinks for relevance and accuracy.
- Validate all article source links.
- Review material content freshness and planned updates.
- Review Core Web Vitals field data where available.
- Confirm robots policy still matches official crawler documentation.
- Confirm no analytics or third-party scripts were introduced.

## Quarterly

- Reverify company number, incorporation status and public company wording.
- Review the full fact ledger and public facts output.
- Confirm professional name, role, biography, portrait and LinkedIn consistency.
- Verify every `sameAs` URL.
- Verify every Yakuji and other external publication link.
- Audit Knowledge Panel status and same-name ambiguity.
- Review official crawler names and purposes.
- Review the 12-month editorial roadmap.
- Revalidate the cross-site entity contract with the corporate website.
- Run the full release suite and representative browser/Lighthouse checks if visible output or dependencies changed.

## Incident triggers

Investigate immediately when:

- the custom domain or HTTPS fails;
- a deployment serves source files rather than `dist`;
- unsupported old copy reappears on the live origin;
- a private document is publicly accessible;
- Search Console reports a manual action or security issue;
- canonical selection moves to the wrong host/path;
- structured data creates a competing Person or Organization;
- an official regulatory source materially changes an essay;
- a public profile falsely claims a licence, customer or capability.

## Evidence template

For every review record:

- date and reviewer;
- source/platform;
- URL or query;
- observed result;
- severity;
- action owner;
- due date;
- resolved date;
- evidence location.

Do not commit private Search Console/Bing exports, account identifiers or email content.
